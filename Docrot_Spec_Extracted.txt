1.  Product  Overview
 
 Documentation  Rot  Detector  is  a  developer  tool  that  automatically  detects  when  project  
documentation
 
becomes
 
outdated
 
as
 
source
 
code
 
evolves.
 
It
 
integrates
 
directly
 
into
 
development
 
workﬂows
 
and
 
continuously
 
scans
 
repositories
 
to
 
identify
 
mismatches
 
between
 
code
 
and
 
documentation.
 
DocRot
 
Detector
 
solves
 
this
 
by
 
analyzing
 
both
 
source
 
code
 
and
 
documentation
 
using
 
static
 
analysis
 
and
 
Abstract
 
Syntax
 
Tree
 
(AST)
 
parsing.
 
The
 
system
 
extracts
 
structured
 
information
 
from
 
code,
 
assigns
 
hashes
 
to
 
code
 
elements,
 
and
 
links
 
those
 
elements
 
to
 
their
 
references
 
in
 
documentation.
 
When
 
code
 
changes
 
in
 
a
 
way
 
that
 
affects
 
documentation,
 
the
 
system
 
automatically
 
ﬂags
 
the
 
affected
 
sections
 
and
 
generates
 
a
 
clear
 
report.
  The  tool  can  run  locally  through  a  command  line  interface  or  automatically  through  Git  hooks  and  CI  
pipelines.
 
In
 
later
 
phases,
 
it
 
can
 
optionally
 
use
 
lightweight
 
AI
 
models
 
to
 
validate
 
mismatches
 
and
 
suggest
 
updated
 
documentation
 
text.
  
1.1  Target  Users   ●  Software  development  teams  maintaining  large  or  fast-moving  repositories  ●  Open-source  maintainers  managing  community  documentation  ●  Technical  writers  responsible  for  API  or  developer  documentation  ●  Engineering  organizations  onboarding  new  developers  to  complex  systems   
1.2  Core  Value  Proposition   ●  Automated  detection  of  outdated  documentation  —  identiﬁes  mismatches  between  code  and  
documentation
 
automatically
 ●  AST-based  code  analysis  —  accurately  extracts  functions,  classes,  parameters,  and  other  code  
elements
 ●  Clear  mismatch  reporting  —  provides  readable  reports  showing  exactly  which  documentation  
sections
 
need
 
review
 ●  CI/CD  integration  —  runs  automatically  on  commits  or  pull  requests  using  Git  hooks  or  GitHub  
Actions
 ●  AI-assisted  suggestions  (Phase  3  )  —  proposes  updated  documentation  wording  or  examples  
when
 
mismatches
 
are
 
detected
 
 
Platform  CLI  scanner  +  Web  dashboard  (React).  Integrates  with  development  workﬂows  and  CI  pipelines.  
Auth  Admin  login  or  token-based  authentication  (MVP).  JWT  sessions  for  dashboard.  Optional  GitHub  OAuth  /  GitHub  App  integration.  
Persistence  PostgreSQL  database.  Stores  projects,  scan  history,  code  elements,  documentation  references,  and  detected  issues.  
Ofﬂine  CLI  scanner  runs  locally  without  internet.  Dashboard  requires  backend  API.   
Environments  Development  (local  CLI  +  backend)  ·  Staging  (Docker  deployment)  ·  Production  (hosted  backend  +  dashboard)  ·  CI/CD  via  GitHub  Actions.

2.  Tech  Stack

Layer  Technology  Notes  /  Constraints  
Frontend  React  19  Web  dashboard:  login,  project  select,  scan  history,  issues  list/detail,  search/ﬁlter.  
Styling  Figma  (design)  +  TailwindCSS  
Keep  UI  clean  and  consistent  with  Figma;  prioritize  readability  for  issue/report  views.  
State  React  state  +  lightweight  client  state  (no  hard  requirement)  
Store  UI  state  (selected  project/scan,  ﬁlters,  search  query);  treat  scans/issues  as  server  data.  
Backend  FastAPI  Serves  dashboard  APIs  and  connects  to  DB;  exposes  scan  +  issue  endpoints.  
Database  PostgreSQL  Persists  projects,  scans,  code  elements,  doc  refs,  issues;  ensure  foreign  keys  by  project_id/scan_id.  
Auth  Simple  admin  login  or  token-based  access  (MVP);  optional  JWT  
MVP  can  be  minimal;  JWT  only  if  needed  for  sessions;  keep  secrets  server-side.  
Validation  FastAPI  +  Pydantic  models  
Validate  all  request  bodies  and  return  consistent  response  schemas  for  the  frontend/CLI.  
Date/Time  Python  datetime  (ISO  8601)  
Store  timestamps  for  scans/issues;  return  ISO  strings  to  frontend.  
Charts  React  chart  library  (optional  for  dashboard)  
Only  needed  if  you  implement  analytics/historical  tracking;  skip  for  MVP  if  time.  
HTTP  Client  Fetch  API  or  Axios  Centralize  API  calls  from  the  dashboard;  include  auth  token  on  requests.  
AI  OpenAI  AI  optional;  used  only  for  validation  +  suggestion  generation  after  deterministic  mismatch  detection.  
Testing  Pytest  (backend/scanner)  +  frontend  tests  as  needed  
Prioritize  unit  tests  for  AST  extraction,  hashing,  and  mismatch  rules;  add  API  tests  for  core  endpoints.  
Hosting  GitHub  Actions  +  AWS  +  Docker  
GitHub  Actions  runs  scans  on  commit/PR;  Docker  for  backend  deployment;  AWS  for  hosting  services.

11.  Non-Functional  Requirements
 
11.1  Correctness  Correctness  is  the  most  critical  non-functional  property  of  this  tool.  A  false  positive  (ﬂagging  
documentation
 
that
 
does
 
not
 
need
 
review)
 
erodes
 
developer
 
trust.
 
A
 
false
 
negative
 
(missing
 
a
 
real
 
semantic
 
change)
 
defeats
 
the
 
tool's
 
purpose
 
entirely.
   
Requirement  Rationale  
Fingerprint  hashes  must  be  fully  deterministic.  Identical  source  code  scanned  twice  —  in  any  order,  on  any  OS,  across  runs  —  must  produce  the  exact  same  hash.  
Non-determinism  would  produce  spurious  ChangeEvents  on  every  run,  making  the  tool  unusable.  
Docstring-only  edits  must  produce  zero  ChangeEvents.  The  normalize_function_ast()  step  must  strip  the  leading  docstring  before  any  feature  is  extracted  or  hashed.  
Developers  routinely  improve  docstrings.  Flagging  doc  changes  as  doc  rot  would  be  self-contradictory.  
Comment-only  and  formatting-only  edits  (whitespace,  blank  lines)  must  produce  zero  ChangeEvents.  
These  are  not  semantic  changes.  The  AST  representation  is  already  whitespace-independent;  no  additional  normalization  is  required.  

===== PAGE 29 =====
A  function  whose  hash  is  unchanged  must  never  reach  the  feature-level  diff  path.  The  hash  fast-path  in  compare_ﬁle_functions()  is  mandatory,  not  optional.  
Performance  and  correctness  both  depend  on  skipping  unchanged  functions.  
Failed  ﬁle  reads  during  a  scan  must  not  remove  or  overwrite  the  stored  baseline  ﬁngerprint  for  that  ﬁle.  
An  I/O  error  mid-scan  must  not  silently  make  every  function  in  that  ﬁle  appear  'removed',  triggering  false  critical  alerts.  
Public  function  additions  and  removals  must  always  be  scored  as  critical  (score=5,  critical=True),  regardless  of  threshold  conﬁguration.  
Adding  or  removing  a  public  API  is  always  a  documentation-relevant  event.

Term  Deﬁnition  
Documentation  Rot  A  state  where  documentation  references  code  behavior,  signatures,  or  features  that  no  longer  match  the  current  source  code.  
Scan  A  single  execution  of  the  DocRot  analysis  process  against  a  repository,  producing  detected  issues  and  reports.  
Project  A  repository  registered  in  the  system  for  documentation  rot  monitoring.  Stored  in  the  Projects  table.  
Code  Element  A  parsed  program  structure  extracted  from  AST  analysis,  such  as  a  function,  method,  or  class.  Stored  in  the  CodeElements  table.  
Fingerprint  A  deterministic  hash  generated  from  structural  code  features  (signature,  control  ﬂow,  conditions,  calls,  side  effects,  exceptions,  returns).  Used  to  detect  semantic  changes.  
Stable  ID  Unique  identiﬁer  for  a  function  or  code  element  generated  as  ﬁle_path/function_name  using  forward  slashes.  Used  as  a  key  across  scans.  
ChangeEvent  A  detected  change  between  two  scans  indicating  that  a  function’s  ﬁngerprint  has  changed,  been  added,  or  been  removed.  
Documentation  Reference  
A  detected  reference  in  documentation  that  points  to  a  code  element  (e.g.,  function_name(),  class  name,  API  endpoint).  Stored  in  DocumentationRefs.  
Issue  A  detected  documentation  mismatch  produced  when  a  code  change  affects  referenced  documentation.  Stored  in  the  Issues  table.  
Critical  Issue  A  high-severity  issue  triggered  by  signiﬁcant  API  changes  such  as  public  function  addition,  removal,  or  major  semantic  modiﬁcation.  
Rot  Score  A  numeric  measure  of  cumulative  documentation  mismatch  severity  produced  during  a  scan.  Stored  in  Scans.rot_score.  
Baseline  The  stored  ﬁngerprint  state  of  a  repository  from  the  previous  scan  used  to  compare  against  the  current  run.  

===== PAGE 35 =====
Report  A  structured  output  artifact  describing  detected  documentation  rot,  generated  in  .docrot-report.json  or  .docrot-report.txt.  
AI  Suggestion  An  optional  AI-generated  recommendation  describing  how  documentation  should  be  updated  to  resolve  an  issue.  Stored  in  AISuggestions.  
CLI  Scanner  The  command-line  tool  (run.py)  that  scans  a  repository,  performs  AST  analysis,  compares  ﬁngerprints,  and  generates  reports.  
Dashboard  The  React  web  interface  used  to  view  projects,  scans,  issues,  and  reports  from  the  backend  API.  
CI  Scan  A  scan  triggered  automatically  by  a  CI/CD  pipeline  (e.g.,  GitHub  Actions)  when  code  changes  are  pushed  or  pull  requests  are  opened.  
First  Run  The  initial  scan  of  a  repository  where  no  baseline  ﬁngerprints  exist.  Fingerprints  are  stored  but  no  issues  are  emitted.  
Clean  Run  A  scan  where  no  ChangeEvents  or  documentation  mismatches  are  detected  compared  to  the  previous  baseline.  
Suggestion  A  single-line  actionable  recommendation  attached  to  a  detected  issue  explaining  how  documentation  should  be  updated.
